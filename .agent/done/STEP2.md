# Step 2: タスク変更処理の統合・詳細編集・一覧拡張

## 目的
当日ファイル `tasks/YYYY-MM-DD.md` に対する変更処理（追加・編集・削除・ステータス更新）を1つの更新系アクションに統合し、
ユーザーが朝/昼/夕方いつ依頼しても同じ仕組みで「今日のタスク」を更新できる運用にする。
また、Step 1で付与した `localTaskId` を使って対象タスクを安定的に同定し、後続の外部同期に耐える更新基盤にする。

## 固定方針
- 意図判定は引き続きLLMで行い、正規表現や固定if分岐で判定しない
- `save_tasks` と `edit_tasks` は分離せず、単一の更新系アクションに統一する
- 当日タスクファイルの変更内容（追加・編集・削除・status更新・detail更新）は単一LLMで判断する
- `modify_tasks` 用LLMは変更差分JSONではなく、更新後の `tasks/YYYY-MM-DD.md` 全文をそのまま出力する
- ユーザー文をそのまま保存せず、要点を簡潔に要約して `tasks/YYYY-MM-DD.md` に反映する
- `list_tasks` では再要約せず、保存済み情報をそのまま返す
- このアプリは「今日のタスク」を運用単位とし、更新対象は常に当日ファイルとする
- タスク更新対象の内部同定は `localTaskId` を優先し、タイトル一致のみの更新を避ける

## 今回の成功条件
- 追加依頼と編集依頼を別アクションに分けず、同一アクションで処理できる
- ユーザーの完了報告メッセージで対象タスクの `status` を更新できる
- ユーザーの補足情報メッセージで対象タスクの `detail` を追記・更新・削除できる
- 必要に応じてタスク自体の削除も同一アクションで処理できる
- `list_tasks` の返却内容は `title` / `status` のみとする
- 返却時は追加のLLM整形を行わず、当日ファイルの内容をそのまま利用できる
- 既存タスクの `localTaskId` が維持され、同一タスクへの連続更新でIDが変わらない

## 実装スコープ（Step 2）
1. アクション構成の統合
- 更新系アクションを1つに統一する
- 想定アクション:
  - `modify_tasks`（追加・編集・削除・ステータス更新・詳細更新を内包）
  - `list_tasks`
  - `others`
- 第1LLMはユーザー入力を `modify_tasks` / `list_tasks` / `others` のいずれかに分類する

2. 変更後ファイル全文の生成（単一LLM, `modify_tasks` のみ）
- 第1LLMの結果が `modify_tasks` のときのみ、変更内容を作成するLLMを呼ぶ
- 変更LLMは以下を判断して、更新後の当日ファイル全文を生成する:
  - どのタスクを残すか
  - 新規追加か、既存更新か、削除か
  - `status` をどう変更するか（例: `todo` -> `done`）
  - `detail` を追記するか、既存内容を更新するか、不要情報を削除するか
- 保存内容は「ユーザー原文の転載」ではなく「要点の要約」にする
- 既存タスク更新時は `localTaskId` を維持した全文更新を優先する

3. 当日タスクファイル更新ルール（`tasks/YYYY-MM-DD.md`）
- 更新対象は常に当日ファイルとする
- 日付が変わったら新しい `tasks/YYYY-MM-DD.md` を対象に運用する（過去日付ファイルは保持）
- 変更種別:
-  - 追加: 新規タスクを含めた全文を再出力
-  - 更新: `title` / `status` / `detail` を反映した全文を再出力
-  - 削除: 不要タスクを除いた全文を再出力
- 既存タスクの基本構造（`id`, `userId`, `title`, `detail`, `status`）をMarkdown中に維持する
- `id`（`localTaskId`）は作成時のみ採番し、更新時は不変とする

4. 一覧表示（`list_tasks`）の拡張
- 当日ファイル `tasks/YYYY-MM-DD.md` から読み取ったタスクを、以下の情報付きで返す:
  - `title`
  - `status`
  - `detail`（存在する場合）
- この返却ではLLMによる再要約・再編集は行わず、保存済み情報をそのまま出力する
- 内部管理用の `id` はMarkdownには保持するが、ユーザー返却に必須表示しない（内部参照に使用）

5. 応答方針（Step 2）
- `modify_tasks`: 変更結果を簡潔な定型メッセージで返す（例: 「今日のタスクを更新しました」）
- `list_tasks`: 当日ファイル `tasks/YYYY-MM-DD.md` の現在値を列挙して返す
- `others`: Step 1の方針を維持

## 変更判断ガイド（LLM向け）
- 優先順位:
  1. 変更意図（追加/更新/削除）を先に確定
  2. 次に `status` 変更の有無を判断
  3. 補足条件（期限・工数・制約）を `detail` に要約反映
- `detail` 記述方針:
  - 簡潔・事実ベース・重複回避
  - 長文を避け、一覧表示で読める粒度を維持
- 削除判断:
  - ユーザーが不要化を明示した場合は削除
  - 矛盾する古い情報は、保持より整合性を優先して整理する
- 対象同定:
  - 内部的には `localTaskId` でタスクを特定する
  - ユーザー入力上で曖昧性が高い場合は、誤更新を避けるため確認応答を優先する

## 受け入れテスト（最小）
1. 朝の追加依頼で `modify_tasks` が選ばれ、当日ファイルへ新規タスクが保存される
2. 昼/夕方の追加依頼でも同じ `modify_tasks` で追記できる
3. 追加された各タスクに `id`（`localTaskId`）が保持される
4. 「Aは終わった」で同一タスクの `status` が更新される（IDは不変）
5. 「Bは2時間、今日18時まで」で同一タスクの `detail` が要約反映される（IDは不変）
6. 「Cは不要」で対象タスクが削除される
7. `list_tasks` 実行時に `title` + `status` が確認できる
8. 更新系入力で `list_tasks` / `others` を誤実行しない
9. 正規表現ベースの意図判定ロジックを導入しない
10. 日付変更後は新しい当日ファイル `tasks/YYYY-MM-DD.md` が更新対象になる
