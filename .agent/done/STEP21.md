# STEP21: 独立した memory アクションの長期記憶探索エージェントを導入する

## 目的

Google Drive 上の `memory/` フォルダを読み取り専用の長期記憶として扱い、ユーザー入力が記憶参照を必要とするときに `memory` という独立アクションを起動し、必要なノードだけを段階的に探索して、ユーザーへの最終返答を生成できるようにする。

## 背景

- 現在の秘書は `modify_events`、`list_events`、`edit_soul`、`edit_user`、`others` をアクションとして持っている
- 新たに `memory/index.md`、`node-registry.yaml`、各種ノード Markdown を持つ長期記憶ストアが追加された
- この記憶はユーザー理解や応答精度を上げる一方、毎回フルスキャンすると API コスト、待ち時間、トークン量が悪化する
- 今回必要なのは記憶の編集ではなく、記憶の必要判定、探索、最終返答生成である
- 現段階では `modify_events` など他アクションへの記憶注入は行わず、`memory` アクション単体で閉じたフローとして実装したい

## 固定方針

1. 今回のスコープでは `memory/` は完全に read-only とし、既存ノード更新や新規ノード作成は行わない。
2. アクション分類器に `memory` を追加し、記憶参照が主目的の質問や確認では `memory` を返す。
3. `memory` アクションの中で、現在は 2 段探索を前提にする。
4. 第 1 段は `index.md` と `node-registry.yaml` を読み、ユーザー質問に対する主要候補ノードを選ぶ。
5. 第 2 段は主要候補ノードの本文まで実際に読んだうえで、どのリンク先を追加で読むべきかを LLM に判断させ、二次候補ノードを選ぶ。
6. 最終段では、主要候補ノードと二次候補ノードの内容をまとめて読み、ユーザー質問への返答を 1 本生成する。
7. 最終返答生成では、根拠が足りない場合に推測で埋めないことを明示し、情報がない場合は「見つかりませんでした」「わかりません」と返す。
8. 探索は有限に止める。初期版では primary 3 ノード、secondary 3 ノード、1-hop までを上限とし、無制限再帰は行わない。
9. 将来は多段探索や外部エージェント・外部 API 連携に拡張できるよう、`memoryAgent` を独立サービスとして設計する。
10. 初期導入対象は `processUserMessage` のみとし、朝夜ジョブや他アクションとの接続は今回の必須スコープから外す。

## 目標アーキテクチャ

### 1. アクション判定

- 既存 `ActionClassifier` の `action` 候補へ `memory` を追加する
- 記憶参照が主目的の入力では `memory` を返し、他アクションとは分岐する
- 初期版では、分類器は `memory` を選ぶだけに留め、探索計画の詳細は memory エージェント側で扱う

### 2. 第1段探索: 主要候補ノードの抽出

- `memory/index.md` を起点に構造を理解する
- `node-registry.yaml` を索引として使い、`id`、`name`、`aliases`、`description`、`type`、`path` を材料に主要候補ノードを絞る
- ここでは `index.md` と `node-registry.yaml` を入力にした候補選定用 LLM を使い、primary 候補を最大 3 ノード返させる
- 出力には少なくとも `node id`、`path`、`reason` を含める

### 3. 第2段探索: 本文を読んだうえで二次候補ノードを選ぶ

- 第1段で選ばれた primary ノードを Google Drive から実際に取得し、frontmatter だけでなく本文も読む
- 本文中の記述と `links` セクションを見て、どの関連ノードがユーザー質問に有効かを判断する
- この判断はリンク選定用 LLM に任せ、secondary 候補を最大 3 ノード返させる
- 「全リンクを読む」のではなく、「本文を踏まえて必要なリンクだけ選ぶ」ことを固定する
- 二次候補は 1-hop のみとし、さらに深い再帰探索は今回行わない

### 4. 最終返答生成

- primary と secondary のノード内容をそろえたうえで、最終返答用 LLM を呼ぶ
- この LLM は、ユーザー質問に対する自然な日本語の返答本文だけを返す
- 返答ルールを強く固定する
  - 読み取れた事実だけを使う
  - 根拠が足りないことは断定しない
  - 情報が見つからない場合は、その旨を明確に言う
  - ありもしない情報を補完しない
- 必要なら、参照したノード名を自然に触れてよいが、メタ説明は最小限にする

### 5. 将来拡張の境界

- 将来は 3 層目、4 層目の探索や、他エージェント・外部 API を使う探索へ拡張できる構造にする
- そのため、`memoryAgent` の中身は「探索段階の配列」を順に実行できる形に寄せる
- ただし今回の実装では、primary 選定、secondary 選定、最終返答生成の 3 LLM 呼び出しで打ち切る

## 実装スコープ

1. Google Drive 読み取り層を拡張し、ルート配下の任意サブフォルダと相対パスのテキストファイルを読める共通関数を追加する
2. `memory/index.md` と `node-registry.yaml` を読む `memoryStore` を追加する
3. `node-registry.yaml` と Markdown frontmatter を安全に parse するため、軽量 YAML 依存追加または最小限 parser を導入する
4. primary 候補選定、secondary 候補選定、最終返答生成を担う `memoryAgent` サービスを追加する
5. `assistantEngine` の action schema / prompt / main flow を拡張し、`memory` アクション時のみ `memoryAgent` を呼ぶ
6. `memory` 専用の prompt を追加し、候補選定と最終返答の hallucination guard を明文化する
7. 記憶取得失敗時のログとフォールバック方針を追加する
8. 単体テストを追加する
   - `memory` が action 候補に含まれること
   - registry の alias / name マッチ
   - primary 3 / secondary 3 / 1-hop 制限
   - secondary 選定が primary ノード本文を前提に行われること
   - 情報不足時に「わかりません」「見つかりませんでした」方向へ返す prompt 制約

## 実装順

1. Drive から `memory/` 配下を読むための I/O とキャッシュ境界を整える
2. `index.md` と `node-registry.yaml` を解決する記憶ストアを作る
3. primary 候補選定 LLM と secondary 候補選定 LLM の入出力スキーマを固める
4. primary ノード本文を読んだうえで secondary を決める 1-hop 展開ロジックを実装する
5. 最終返答生成 LLM と hallucination guard を実装する
6. `assistantEngine` の分類器 schema と prompt を拡張し、`memory` アクション分岐を追加する
7. テストと README を更新する

## 成功条件

- ユーザーが大学、友人、所属組織、過去イベント、継続プロジェクトなどの記憶参照質問をしたとき、`memory` アクションが選ばれ、必要な記憶だけ探索される
- `memory` 以外のアクションでは、今回追加した長期記憶探索フローは起動しない
- 記憶探索は有限で止まり、無制限なリンク再帰を起こさない
- `memory/` の内容は一切更新しない
- 情報不足時に、存在しない事実を補って答えず、「見つかりませんでした」「わかりません」と返せる
- 既存の `modify_events`、`list_events`、朝夜ジョブの動作を壊さない
- `.agent/README.md` の索引が今回の進行中プランを反映している

## 未解決だが今回の実装内で判断すること

- `memory/` フォルダ名を固定値にするか、環境変数で切り替え可能にするか
- frontmatter と `node-registry.yaml` の parser を依存追加で行うか、自前実装にするか
- primary 候補選定で index と registry のどこまで全文を LLM に渡すか、事前整形を入れるか
- 最終返答時に参照ソース一覧を内部保持のみにするか、必要なら返答へ軽く露出させるか
