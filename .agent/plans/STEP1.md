# Step 1: 非定型タスク入力のLLM処理基盤（Google Drive保存）

## 目的
毎朝の「朝です」への返信として送られる自由形式テキストを、
フォーマット依存の分岐や正規表現ルールを使わずにLLMで理解し、
Google Drive上の `tasks.md` にタスクとして保存・参照できる状態を作る。

## 固定方針
- 定型的なif分岐・パターンマッチ・正規表現ベースの意図判定は実装しない
- ユーザー入力の解釈はLLMに一任する
- 失敗時のフォールバックはユーザーへのエラーメッセージ返却のみ
- 改善ポイントは「LLMに渡すコンテキスト設計」で解決する

## 今回の成功条件
- ユーザーの自由形式メッセージ（改行/句読点区切り/時間ありなし/詳細ありなし）からタスクが保存される
- 保存先はローカルJSONではなく、Google Driveフォルダ直下の `tasks.md`
- ユーザーが自然文で「今日のタスクは何？」「今日は何をすればいい？」等を聞いた時に、`tasks.md` から現在タスク一覧を返せる
- 上記判定は定型ロジックではなくLLMの解釈で行う

## 実装スコープ（Step 1）
1. 入力受信
- LINE webhookで受け取ったテキストをそのままLLMに渡す

2. LLMアクション生成
- LLMに `action_plan` JSONを生成させる
- 最低限のアクション:
  - `save_tasks`（複数可）
  - `list_tasks`
  - `small_talk`（必要なら）

3. Google Drive永続化
- `.env.example` で定義されるGoogle Drive設定を利用
- 指定フォルダに `tasks.md` を作成（存在しなければ作る）
- `save_tasks` 時に `tasks.md` を更新
- `list_tasks` 時に `tasks.md` を読み込み

4. 応答生成
- 実行結果（保存件数/一覧）をコンテキストとしてLLMへ渡し、自然文レスポンスを生成する

## `tasks.md` 方針
- 人間が読めるMarkdownファイルとする
- LLM処理と整合するよう、タスクデータの構造が崩れない形式で保持する
- 最低保持項目:
  - `id`
  - `userId`
  - `title`
  - `detail`（任意）
  - `estimateMin`（任意）
  - `status`（初期値: `todo`）
  - `createdAt`
  - `updatedAt`

## 受け入れテスト（最小）
1. 「朝です」送信後に自由形式で複数タスクを送る
2. Google Driveの指定フォルダ内 `tasks.md` に保存される
3. 「今日のタスクは何？」で `tasks.md` ベースの一覧が返る
4. 定型判定ロジック（正規表現分岐）がコード上に存在しない
