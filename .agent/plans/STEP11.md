# Step 11: SOUL.md / USER.md 編集アクションの追加

## 目的
既存アクション分類（`modify_tasks` / `list_tasks` / `others`）を拡張し、
`SOUL.md` と `USER.md` を更新できる専用アクションを追加する。
更新処理とユーザー向け結果出力は分離せず、`edit_soul` / `edit_user` 用の
LLMコールが一貫して担当する前提を定義する。

## 固定方針
- Step 11 は仕様定義が主目的であり、機能実装は行わない
- ActionClassifier の分類先を以下の5種に拡張する
  - `modify_tasks`
  - `list_tasks`
  - `edit_soul`
  - `edit_user`
  - `others`
- `edit_soul` / `edit_user` 判定精度を上げるため、ActionClassifier プロンプトを更新する
- `edit_soul` / `edit_user` 実行時は、編集用LLMが更新計画・実行・結果出力を行う

## 対象LLM
- ActionClassifier（分類拡張・プロンプト更新）
- EditSoulAgent（`SOUL.md` 更新の計画・実行・結果出力）
- EditUserAgent（`USER.md` 更新の計画・実行・結果出力）

## ActionClassifier 仕様（Step 11）
1. 出力アクション集合の拡張
- 従来の3分類から5分類へ拡張する

2. 判定ルールの明確化
- `edit_soul`: AIの人格・口調・振る舞いルール（`SOUL.md`）の追加/修正/削除意図
- `edit_user`: ユーザー属性・嗜好・前提情報（`USER.md`）の追加/修正/削除意図
- タスク操作意図は既存どおり `modify_tasks` / `list_tasks` を優先
- どれにも該当しない入力は `others`

3. プロンプト更新要件
- `SOUL.md` 編集依頼と `USER.md` 編集依頼を言い換え含めて判定可能にする
- 「単なる相談/質問」と「実際の編集指示」を分離する基準を明記する
- 既存3アクションとの衝突ケース（例: タスク文脈と混在する指示）の優先順位を明記する

## 編集実行LLM（EditSoulAgent / EditUserAgent）仕様
1. 呼び出しタイミング
- ActionClassifier が `edit_soul` または `edit_user` と判定した時点で呼ぶ

2. 目的
- 対象ファイル更新に向けた変更計画を立てる
- 計画に基づいて更新処理を実行する
- 実行結果としてユーザー向けの変更内容説明を返す

3. 入力コンテキスト
- `SOUL.md` 内容
- `USER.md` 内容
- 当日会話ログ（Step 9の会話履歴）
- ユーザーの当該編集依頼文（最新入力）

4. 出力要件
- 更新対象ファイル名（`SOUL.md` / `USER.md`）を明示する
- 実際に行った変更点を短く列挙または要約する
- 不必要に長文化せず、ユーザーが次アクション判断できる情報量にする

## 今回の成功条件
- ActionClassifier が5アクションを扱う前提で設計されている
- ActionClassifier プロンプト更新要件が明文化されている
- `edit_soul` / `edit_user` 実行時の計画・実行・結果出力フローが定義されている
- 更新通知専用LLMを導入しない方針が明記されている
- 編集実行LLMに当日会話ログ・`SOUL.md`・`USER.md` を渡す要件が明記されている

## 実装スコープ（Step 11）
1. アクション体系拡張仕様の定義
- `edit_soul` / `edit_user` を追加した分類仕様を確定する

2. ActionClassifier プロンプト改訂仕様の定義
- 新規2アクションを安定判定するための指示文要件を定義する

3. 編集実行LLMコール仕様の定義
- 呼び出し条件、入力、出力フォーマット、責務境界を定義する

4. ユーザー応答方針の定義
- 更新成功時は編集実行LLMが生成した結果出力を返す
- 更新失敗時は失敗理由を返し、未実施更新を成功扱いで通知しない

## 受け入れテスト（最小）
1. `SOUL.md` の編集依頼入力が `edit_soul` に分類される想定になっている
2. `USER.md` の編集依頼入力が `edit_user` に分類される想定になっている
3. タスク編集依頼入力は引き続き `modify_tasks` に分類される想定になっている
4. `edit_soul` で編集実行LLMが計画・更新・結果出力まで行う想定になっている
5. `edit_user` で編集実行LLMが計画・更新・結果出力まで行う想定になっている
6. 結果出力に「対象ファイル」と「主な変更点」が含まれる想定になっている

## 非スコープ（Step 11時点）
- `SOUL.md` / `USER.md` 実ファイル更新ロジックの実装
- 変更差分生成アルゴリズムの最適化
- 結果出力テンプレートの最終UIデザイン調整
- 新規アクションのオンライン学習・自動評価基盤の構築

## 完了後の期待効果
- `SOUL.md` / `USER.md` への編集意図を明示的に扱える
- ActionClassifier の意図判定がより実運用に即した粒度になる
- 更新内容が編集処理と同一コンテキストで可視化され、責務分断による齟齬を減らせる
