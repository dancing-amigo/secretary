# Step 2: タスク更新・詳細編集・一覧拡張

## 目的
朝に保存した当日ファイル `tasks-YYYY-MM-DD.md` のタスクを、ユーザーから後続で届くメッセージに応じて更新できるようにする。
特に「終わった」などの進捗報告でステータスを更新し、必要に応じてタスク詳細（所要時間・期限・補足条件など）も編集できる運用を確立する。

## 固定方針
- アクション判定は引き続きLLMで行い、正規表現や固定if分岐で意図判定しない
- Step 1と同様に、責務を分けた複数LLM構成を維持する
- ユーザー文をそのまま保存せず、編集内容はLLMが要点を簡潔に要約して当日ファイル `tasks-YYYY-MM-DD.md` に反映する
- `list_tasks` では再要約せず、当日ファイル `tasks-YYYY-MM-DD.md` に記録済みの情報をそのまま返す
- このアプリは「今日のタスク」を運用単位とし、編集対象は常に当日ファイルとする

## 今回の成功条件
- ユーザーの完了報告メッセージで対象タスクの `status` を更新できる
- ユーザーの補足情報メッセージで対象タスクの `detail` を追記・更新・削除できる
- タスク編集用の新アクションが追加され、既存の `save_tasks` / `list_tasks` / `others` と共存できる
- `list_tasks` の返却内容に `title` だけでなく `status` と `detail` が含まれる
- 返却時は追加のLLM整形を行わず、当日ファイル `tasks-YYYY-MM-DD.md` の内容をそのまま利用できる

## 実装スコープ（Step 2）
1. アクション拡張
- 既存アクションに加えて、タスク編集系アクションを追加する
- 追加対象の想定:
  - `edit_tasks`（ステータス更新・詳細編集を内包）
- 第1LLMはユーザー入力を `save_tasks` / `list_tasks` / `edit_tasks` / `others` のいずれかに分類する

2. 編集計画の生成（第2LLM, `edit_tasks` のみ）
- 第1LLMの結果が `edit_tasks` のときのみ、第2LLMを呼ぶ
- 第2LLMは以下を判断して編集計画を作る:
  - どのタスクを対象にするか
  - `status` をどう変更するか（例: `todo` -> `done`）
  - `detail` を追記するか、既存内容を更新するか、不要情報を削除するか
- 編集内容は「ユーザー原文の転載」ではなく「要点の要約」を保存形式にする

3. 当日タスクファイル更新ルール（`tasks-YYYY-MM-DD.md`）
- 更新対象は常に当日ファイルとする
- 日付が変わったら新しい `tasks-YYYY-MM-DD.md` を対象に運用する（過去日付ファイルは保持）
- ステータス更新:
  - 完了報告が来たら該当タスクの `status` を更新
- 詳細編集:
  - 所要時間、期限、実施条件、進行上の注意点などを `detail` に反映
  - 第2LLMの判断で、追記・置換・削除を選択可能にする
- 既存タスクの構造（`userId`, `title`, `detail`, `status`）は維持する

4. 一覧表示（`list_tasks`）の拡張
- 当日ファイル `tasks-YYYY-MM-DD.md` から読み取ったタスクを、以下の情報付きで返す:
  - `title`
  - `status`
  - `detail`（存在する場合）
- この返却ではLLMによる再要約・再編集は行わず、保存済み情報をそのまま出力する

5. 応答方針（Step 2）
- `edit_tasks`: 更新結果を簡潔な定型メッセージで返す（例: 「タスクを更新しました」）
- `list_tasks`: 当日ファイル `tasks-YYYY-MM-DD.md` の現在値を列挙して返す
- `save_tasks` / `others`: Step 1の方針を維持

## 編集判断のガイド（LLM向け）
- 優先順位:
  1. ステータス変更が明示されている場合は `status` 更新を優先
  2. 補足条件・期限・工数などがあれば `detail` を要約して反映
- `detail` 記述方針:
  - 簡潔・事実ベース・重複回避
  - 長文を避け、後で一覧表示しても読める粒度にする
- 削除判断:
  - 古くなった条件や矛盾する情報は、保持より整合性を優先して整理する

## 受け入れテスト（最小）
1. 朝に保存済みタスクに対して「Aは終わった」で `status` が更新される
2. 「Bは2時間かかる、今日18時まで」などで `detail` が要約反映される
3. 追記後に `list_tasks` を実行すると `title` + `status` + `detail` が確認できる
4. `list_tasks` 返却時に追加のLLM要約をせず、当日ファイル `tasks-YYYY-MM-DD.md` の内容がそのまま出る
5. 編集系入力で `save_tasks` や `others` を誤実行しない
6. 正規表現ベースの意図判定ロジックを導入しない
7. 日付変更後は新しい当日ファイル `tasks-YYYY-MM-DD.md` が編集対象になる
