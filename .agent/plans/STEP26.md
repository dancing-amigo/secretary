# STEP26

## 目的

朝ジョブの入力を再設計し、`summaryContextText` 廃止後は前日分 daily record に加えて直近3日分の `record/timeline/days/YYYY-MM-DD.md` を参照して朝メッセージを生成する。  
朝メッセージの役割は「その日の確定計画を決めること」ではなく、当日の固定予定を踏まえつつ、やるべき候補や持ち越し候補を複数提案して、ユーザーが最終判断しやすい状態を作ることに置く。  
今回の Step では `memory/` は使わず、直近 daily record と当日 Calendar event だけで提案精度を上げる。

## 実装内容

1. 朝ジョブの入力元を `summaryContextText` から切り替え、前日分に加えて直近3日分の `record/timeline/days/YYYY-MM-DD.md` を読み出せる helper を追加する。
2. 朝ジョブの入力コンテキストを、`SOUL.md`、`USER.md`、当日 Calendar event 一覧、前日分 daily record、直近3日分 daily record 群に再定義する。
3. daily record は Markdown 全文をそのまま扱うだけでなく、`## 今日の予定` と `## ノート` に含まれる持ち越し候補、未完了っぽい話題、繰り返し出ている関心事を朝 prompt で読み取れる前提へ説明を更新する。
4. 朝メッセージ生成プロンプトを、挨拶中心の文面生成から「当日の固定予定を確認しつつ、今日やる候補を複数提案する」役割へ変更する。
5. 提案文面の必須要件を、自然な朝の挨拶、当日の固定予定への短い言及、今日やる候補の複数提案、必要なら時間帯や優先順の軽い示唆を含む形に再定義する。
6. 提案は断定的な確定計画ではなく、`これもどうですか` `この前後でこれを進めるのはありです` のような提案口調を必須化し、最終決定はユーザーに委ねる方針を明記する。
7. 当日 Calendar に載っていない項目でも、直近3日の daily record から持ち越しや未着手が読み取れるものは、朝の候補提案に含めてよいルールを追加する。
8. 候補数は固定値で縛らず、朝に読みやすい長さを保ちながら情報量に応じて増減させる。ただし少なすぎて提案性が消えたり、多すぎて読みにくくなったりしないよう prompt で上限感を与える。
9. `memory/` を朝ジョブへ注入する処理、朝用 memory retrieval、長期記憶ベースの候補選定は今回のスコープから外し、関連する文書にも非スコープとして明記する。
10. テストは `直近3日 daily record + 当日予定` を朝ジョブが入力に使うこと、前日 summary に依存しないこと、持ち越し候補を提案に含められること、提案が確定命令調になりすぎないことを検証できる形へ更新する。
11. 実装完了時には、旧来の「前日 summary を短く参照して挨拶するだけ」の前提に依存する補助関数、分岐、テストケース、文書記述が不要なら必ず削除する。

## 完了条件

1. 朝ジョブが `summaryContextText` ではなく、前日分と直近3日分の daily record を入力に使う。
2. 朝ジョブは当日の固定予定を踏まえたうえで、今日やる候補を複数提案できる。
3. 当日 Calendar に存在しない持ち越し候補でも、直近3日 daily record に根拠があれば提案に含められる。
4. 朝メッセージは確定スケジュールの押しつけではなく、ユーザーが選べる提案口調になっている。
5. `memory/` や朝用 retrieval を使わなくても、直近 daily record だけで提案精度を上げる構成になっている。
6. `summaryContextText` 前提の朝ジョブ実装、関連テスト、関連文書の不要部分が整理されている。
7. `npm test` が通る。
