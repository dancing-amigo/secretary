# Step 5: Google Calendar event の読取同期をアプリ行動前に行う

## 目的
Google Calendar event を外部スケジュールの参照先として扱う。
アプリが何か行動する直前に Google Calendar から当日予定を取得してローカルへ反映し、
ユーザーがカレンダー上で追加・編集した内容をアプリ側も理解できる状態にする。

## 固定方針
- Step 5 で扱う外部読取対象は Google Tasks ではなく Google Calendar Events
- 同期タイミングは定期実行ではなく「アプリが処理を始めるたび」とする
- Step 4 のアプリ -> Google Calendar 同期と組み合わせ、Calendar を実質的な共有面にする
- 取得した予定は、後続の判定、提案、LINE 返信生成、タスク編集判断の前提データとして使う
- 同期失敗時もアプリ全体は止めず、取得失敗を明示したうえで安全側の挙動に倒す

## 前提（Step 4完了後）
- `modify_tasks` 後に当日タスク全体を Google Calendar event へ同期する基盤がある
- Google Calendar API を呼ぶ認証情報とカレンダー ID 設定がすでにある
- ローカルには `tasks/YYYY-MM-DD.md`、同期状態には `task-sync-state.json` がある

## 今回の成功条件
- アプリがユーザー入力を受けて処理を始める前に、当日 Google Calendar event を取得できる
- Google Calendar 側で追加された予定を、アプリが当日の文脈として認識できる
- Google Calendar 側で変更された予定内容を、次回アプリ行動時に反映できる
- 予定の重複や空き時間判断に、最新の Calendar 状態を使える
- 競合時はアプリ側の情報 (google drive) を優先する

## 実装スコープ（Step 5）
1. 同期トリガーの定義
- 同期は cron ではなく、各アクション実行前に行う
- 対象例:
  - LINEメッセージ受信直後
  - 朝メッセージ/夜サマリー生成前
- 1 リクエスト中で複数回呼ばないよう、同一処理内では 1 回の同期結果を共有する

2. 予定取得対象の定義
- 取得対象カレンダー（例: `primary`）を設定可能にする
- 基本取得期間は「当日 00:00〜23:59（TZ 準拠）」とする
- 終日予定と時間付き予定をどちらも含める
- 必要なら直近の前後バッファを持たせるが、主対象は当日予定とする

3. ローカル取り込み方針
- Google Calendar event 一覧をローカル実行コンテキストへ取り込む
- Step 4 でアプリから作成した event と、カレンダー上で手動追加された event を区別せず読めるようにする
- 少なくとも後続処理が参照できる共通の「当日予定一覧」を作る
- この段階では tasks ファイルへの自動マージ方法は最小限に留め、まずはアプリが予定を理解できることを優先する

4. マッピングと識別
- Step 4 由来の event は `task-sync-state.json` 上の `googleCalendarEventId` と照合できるようにする
- カレンダー上で新規作成された event も、未対応の外部予定として識別できるようにする
- 将来的な重複判定や編集提案のため、`eventId`、時間帯、タイトルなど最低限の識別子を保持する

5. 利用箇所
- ActionClassifier や後続エージェントが、その時点の予定を見たうえで判断できるようにする
- 予定重複チェック、空き時間推定、ユーザーがカレンダーで追加した予定の考慮を可能にする
- 今後、LINE 上の編集提案や再計画にこの予定一覧を流用できる形にする

6. エラー処理
- Calendar 読取に失敗しても受信処理自体は継続する
- 失敗時はログに残し、後続には「最新カレンダー同期なし」であることが分かる状態を渡す
- 同一エラーが継続した際に追跡できるよう、対象カレンダー、期間、操作名を記録する

## 受け入れテスト（最小）
1. ユーザーがアプリへメッセージを送ると、その処理開始前に当日 Google Calendar event が取得される
2. カレンダー上で手動追加した予定を、次回アプリ行動時に認識できる
3. カレンダー上で変更した時間帯やタイトルが、次回アプリ行動時の判断材料に反映される
4. 同一リクエスト中に不要な多重同期が走らない
5. Calendar API 取得失敗時もアプリは停止せず、失敗ログだけが残る

## 非スコープ（Step 5時点）
- Google Tasks 連携の再導入
- 定期ポーリング同期
- Google Calendar event を即座に `tasks/YYYY-MM-DD.md` へ全面反映する高度なマージ
- 繰り返し予定、参加者、会議URL などの高度な属性解釈

## 完了後の期待効果
- ユーザーがカレンダー上で加えた予定変更を、アプリが行動前に毎回踏まえられる
- 予定重複や空き時間を見たうえで、より現実的な提案や編集判断ができる
- 外部スケジュールの参照先を Google Calendar に一本化できる
